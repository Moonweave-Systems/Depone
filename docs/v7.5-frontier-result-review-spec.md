# V7.5 Frontier Result Review Spec

Status: first slice implemented
Date: 2026-06-15

## Purpose

V7.5 reviews V7 frontier-result evidence before any runtime ingestion treats the
next phase as complete. V7 proves an allowlisted fixture produced
`release-decision.md`. V7.5 decides whether that output satisfies the
`release_decision` packet contract.

This slice closes the second-loop trust gap:

```text
V6 frontier
-> V6.5 frontier dispatch
-> V7 controlled frontier result
-> V7.5 reviewed frontier result
-> future V6-style ingestion
```

The output is still not workflow advancement. It is a reviewed result artifact
that a later runtime-ingestion slice can consume.

## Product Position

V7.5 is the frontier-result counterpart to V5.5. V5.5 reviews a scheduled worker
result from the first post-entry phase. V7.5 reviews a later frontier result
after the runtime has looped through V6 and V6.5.

The review contract must be deterministic, hash-bound, and resumable. A
frontier result is not trusted because V7 produced a file; it is trusted only
when the review can tie the produced output back to the V6 frontier packet, V6.5
dispatch bundle, and V7 result hashes.

## Command Contract

```bash
python scripts/review_frontier_result.py --result out/v7/<run_id> --out out/v7.5/<run_id>
python scripts/review_frontier_result.py --resume out/v7.5/<run_id>
python scripts/review_frontier_result.py --self-test
```

## Accepted Inputs

V7.5 accepts only an owned V7 result directory with:

- `.run_frontier_result-owned.json`,
- `status.json` with `status: executed`,
- `result.json`,
- `hashes.json`,
- `stdout.txt`,
- `stderr.txt`,
- produced outputs under `work/`,
- source dispatch hashes that still validate through V6.5 resume.

V7.5 rejects:

- missing or symlinked files,
- stale V6.5 dispatch source,
- tampered V7 result, hashes, stdout, stderr, or produced outputs,
- `failed`, `blocked`, or `invalid` V7 statuses,
- results with no produced outputs,
- outputs outside owned `out/v7/<run_id>/work/`.

- `review_id`,
- `verdict`,
- `source_result_path`,
- `source_packet_id`,
- `source_phase_id`,
- `source_hashes`,
- `findings`,
- `approved_outputs`.

## Verdicts

- `approve`: output satisfies the packet's expected output contract.
- `request-changes`: output exists but does not satisfy expected output checks.
- `needs-human`: deterministic review cannot decide safely.
- `invalid`: source evidence or review artifacts are malformed or stale.

The first fixture review may approve only `release-decision.md` for the
`v6-frontier-0001-release_decision` dogfood packet. It should not generalize to
later `release_decision` packets or arbitrary output schemas yet.

## Review Rules

For the first slice, `approve` requires:

1. V7 status is `executed`.
2. V7 resume is `resumable`.
3. `result.json`, `packet.json`, and `dispatch.json` identify
   `v6-frontier-0001-release_decision` / `release_decision`.
4. `release-decision.md` is listed in `produced_outputs`.
5. `work/release-decision.md` exists and is non-empty.
6. The output hash matches V7 `hashes.json`.
7. The source packet expected outputs include `release-decision.md`.
8. The source packet and dispatch stop conditions require V7.5 frontier-result
   review before runtime ingestion.

Any hash or ownership failure is `invalid`, not `request-changes`.

## Resume Behavior

`--resume` recomputes the source V7 result, source hashes, review body,
markdown, and review hashes. It writes only `status.json` and `resume.md`.

Resume returns:

- `review-approved` with `resume_state: resumable` when everything matches,
- `changes-requested` when deterministic output checks fail,
- `needs-human` when the source is valid but outside the fixture review scope,
- `invalid` when source or review artifacts are stale or malformed.

## Non-Goals

- Do not advance or ingest runtime state.
- Do not execute workers.
- Do not repair frontier outputs.
- Do not merge outputs into the repository.
- Do not interpret arbitrary output schemas.
- Do not call Codex CLI, OMX, network APIs, paid APIs, or external services.
- Do not commit, push, install dependencies, deploy, access secrets, delete
  files, rewrite history, or send external messages.

## Fixtures

The first slice should cover:

- approve `out/v7/v32-semantic-dogfood`,
- reject missing `release-decision.md`,
- invalidate tampered output hash,
- invalidate tampered review artifact,
- clean resume,
- stale source result invalidation,
- needs-human for unsupported phase or output schema.

## Release Criteria

The slice is `keep` only if:

- `python scripts/review_frontier_result.py --self-test` passes,
- dogfood review over `out/v7/v32-semantic-dogfood` returns
  `review-approved`,
- clean resume is resumable,
- tampered review and tampered source evidence invalidate resume,
- no runtime ingestion or worker execution is introduced.

## Next Slice

After V7.5, the next slice should ingest reviewed V7.5 frontier results into the
runtime state, completing the second loop without trusting unreviewed V7 output.
