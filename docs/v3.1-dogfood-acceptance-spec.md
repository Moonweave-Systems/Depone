# V3.1 Dogfood Acceptance Spec

Status: planned dogfood workflow
Date: 2026-06-15

## Purpose

V3.1 tests whether the current control plane is useful on this repository
itself before adding V4 parallel orchestration. The goal is not to add another
runtime feature. The goal is to prove, with a real repo-scoped workflow, whether
V0.5 through V3 can produce a next packet that a human or agent can actually
use.

The dogfood target is release readiness for the current V2, V2.5, and V3 work.
This is intentionally close to the live repo state: many source files are
changed or untracked, release evidence exists under ignored `out/`, and the
next useful action is deciding whether the current implementation is ready to
commit, adjust, or defer.

## Non-Goals

- Do not implement V4 parallel orchestration.
- Do not execute arbitrary shell commands through the public V2 interface.
- Do not claim fully autonomous large-task completion.
- Do not commit, push, merge, delete, install dependencies, deploy, access
  secrets, or send external messages.

## Workflow Under Test

The dogfood workflow uses the current public control plane:

1. validate a real `workflow.plan.json` against the V0.5 contract,
2. compile it with V1 into a first-slice packet,
3. execute one manifest-scoped V2 local-shell attempt,
4. review the V2 attempt with V2.5,
5. advance with V3 into a deterministic next packet,
6. evaluate the next packet for practical usefulness.

Because the public V2 command surface intentionally refuses arbitrary commands,
the V2 attempt proves controlled execution and artifact-backed automatic
verification, not semantic completion of the whole release-readiness task. The
acceptance decision must account for that limitation.

## Dogfood Plan

The tracked plan is
`fixtures/v3.1/dogfood-release-readiness.workflow.plan.json`.

The workflow has four phases:

1. `release_inventory`: inspect the repository release surface and produce
   `inventory.json`.
2. `evidence_review`: verify the inventory against generated summaries,
   contract checks, and tracked docs, producing `verification.md`.
3. `release_decision`: decide keep, adjust, or defer for the current V2/V2.5/V3
   release candidate, producing `release-decision.md`.
4. `human_gate`: record human approval or deferral, producing
   `human-approval.md`.

The first slice is `release_inventory`. V3 should therefore advance to
`evidence_review`.

## Acceptance Criteria

The dogfood workflow is `keep` only if all of these are true:

- the dogfood plan passes `scripts/evaluate_plan.py`;
- V1 compiles the dogfood plan into a ready first-slice packet;
- V2 records a verified automatic attempt with artifact-backed checked state;
- V2.5 records `review-approved`;
- V3 records `advanced`;
- V3 `next/0001.packet.json` selects `evidence_review`;
- the next packet contains enough inputs, expected outputs, workers, and stop
  conditions for another Codex agent to continue without reading the whole
  repo;
- the report clearly states whether current V2 restrictions prevent stronger
  real-world proof.

The decision is `adjust` if the pipeline runs but the next packet is too vague,
too ceremony-heavy, or not clearly actionable.

The decision is `defer` if the public command surface prevents meaningful
dogfood evidence.

## Evaluation Rubric

The dogfood report must score these dimensions as `pass`, `weak`, or `fail`:

- plan realism,
- compile fidelity,
- execution evidence,
- review trust,
- runtime advancement,
- next-packet usefulness,
- ceremony cost,
- release-readiness signal.

## Required Commands

```bash
python scripts/evaluate_plan.py --plan fixtures/v3.1/dogfood-release-readiness.workflow.plan.json
python scripts/compile_workflow.py --plan fixtures/v3.1/dogfood-release-readiness.workflow.plan.json --out out/v1/v31-dogfood-release-readiness --mode compile
python scripts/execute_packet.py --manifest fixtures/v2/manifest.json --out out/v2/v31-dogfood-manifest
python scripts/execute_packet.py --review out/v1/v2-v31-dogfood-manifest-dogfood-v31-inventory-verified --out out/v2/v31-dogfood-manifest/dogfood-v31-inventory-verified
python scripts/run_workflow.py --start out/v2/v31-dogfood-manifest/dogfood-v31-inventory-verified --out out/v3/v31-dogfood-release-readiness
python scripts/run_workflow.py --resume out/v3/v31-dogfood-release-readiness
```

The V2 command is intentionally manifest-scoped. The trusted manifest contains
the dogfood fixture and uses only allowlisted Python snippets.

## Decision Output

The report is `docs/v3.1-dogfood-report.md`. It must include:

- exact commands run,
- generated status summary,
- next packet phase and expected outputs,
- rubric scores,
- keep/adjust/defer decision,
- recommended next workflow.
