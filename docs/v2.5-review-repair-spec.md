# V2.5 Execute-Review-Repair Spec

Status: implemented first loop
Date: 2026-06-15

## Purpose

V2 executes or prepares exactly one trusted V1 first-slice packet and records
evidence. V2.5 adds one bounded review and repair loop around that evidence
without becoming a multi-slice runtime.

The goal is to turn an executed packet into a reviewed packet state:

- preserve original execution evidence,
- let independent reviewers inspect the evidence and current diff,
- store review findings as structured artifacts,
- allow at most a bounded repair attempt when trust and gates still allow it,
- derive final packet status from execution, verification, review, and repair
  evidence.

V2.5 still does not advance to later workflow phases. V3 owns packet
advancement.

## Product Position

| Layer | Responsibility | V2.5 stance |
| --- | --- | --- |
| V1 compiler | deterministic packet and gate input | required |
| V2 adapter | execute one packet and capture evidence | required |
| V2.5 loop | review evidence and optionally repair one packet | first loop implemented |
| V3 runtime | select and advance later packets | defer |

V2.5 is the quality gate between "a packet ran" and "the system may consider
the packet useful input for later runtime advancement."

## Non-Goals

- Do not select or compile later packets.
- Do not merge, push, deploy, install dependencies, access secrets, send
  external messages, rewrite history, or delete files.
- Do not accept a reviewer's prose as proof without structured findings and
  evidence.
- Do not run unbounded repair loops.
- Do not repair a packet whose V1/V2 trust checks are stale or malformed.
- Do not hide failed repairs by overwriting prior attempts.

## Command Contract

V2.5 extends the V2 adapter with stdlib-only modes while keeping V2 behavior
stable:

```bash
python scripts/execute_packet.py --review out/v1/<run_id> --out out/v2/<run_id>
python scripts/execute_packet.py --repair out/v1/<run_id> --out out/v2/<run_id>
python scripts/execute_packet.py --review-resume out/v1/<run_id> --out out/v2/<run_id>
python scripts/execute_packet.py --manifest fixtures/v2.5/manifest.json --out out/v2.5/<suite_id>
```

Mode semantics:

- `--review`: inspect trusted V2 attempts and write deterministic review
  artifacts only.
- `--repair`: prepare one bounded repair prompt when review findings are
  actionable, trust checks pass, and risk gates allow the requested work.
- `--review-resume`: recompute V1 trust, V2 evidence trust, review artifact
  hashes, repair artifact hashes, and final packet status; write only status and
  resume files.
- `--manifest`: run deterministic V2.5 fixtures. Public manifest execution must
  be limited to the checked-in `fixtures/v2.5/manifest.json`.

## Evidence Model

V2.5 adds root-level artifacts beside V2 attempts:

```text
out/v2/<run_id>/
├── reviews/
│   └── 0000/
│       ├── review.json
│       ├── review.md
│       └── hashes.json
├── repairs/
│   └── 0000/
│       ├── repair-attempt.json
│       ├── prompt.md
│       ├── verification.json
│       └── hashes.json
├── review-contracts.json
├── repair-contracts.json
├── status.json
└── resume.md
```

`review-contracts.json` and `repair-contracts.json` are parent-level trust
anchors, following the V2 `attempt-contracts.json` pattern. Coherent rewrites
inside a review or repair directory must invalidate resume.

## Review Artifact Contract

Each `review.json` must include:

- `review_id`: zero-based 4-digit ID,
- `source_attempt_id`: V2 attempt being reviewed,
- `reviewer_role`,
- `scope`,
- `findings`: list of structured findings,
- `summary`,
- `verdict`: `approve`, `request_changes`, or `needs_human_review`,
- `created_at`,
- `source_hashes`: packet, prompt, attempt contract, and V2 status hashes.

Each finding must include:

- `finding_id`,
- `severity`: `critical`, `warning`, or `note`,
- `file_or_artifact`,
- `line_or_pointer`,
- `claim`,
- `evidence`,
- `falsifier`,
- `suggested_fix`,
- `repair_allowed`: boolean.

Findings without evidence remain `needs_human_review`; they cannot authorize
repair.

## Repair Policy

A repair attempt may run only when all are true:

1. V1 resume is trusted.
2. V2 evidence is trusted.
3. Latest review verdict is `request_changes`.
4. At least one finding has `repair_allowed: true`.
5. Required risk gates are not blocked.
6. First-loop repair prompt preparation does not launch a backend.
7. Retry count is below the configured cap.

Repair writes a new repair artifact; it never edits the original V2 attempt,
review, or contract ledger. The implemented first loop stops at
`repair-prepared`; backend repair execution is deferred to a later slice.

V2.5 should default to `needs-human` instead of repair when a finding implies
dependency install, production deploy, secret access, external messaging,
database migration, public API change, deletion, or history rewrite.

## Status Model

V2.5 packet status values:

- `review-pending`: V2 attempt exists but no review exists.
- `review-approved`: latest trusted review approves the attempt.
- `changes-requested`: latest trusted review has actionable findings.
- `repair-prepared`: repair prompt/evidence was prepared but no backend ran.
- `repair-executed`: repair backend exited successfully but verification is not
  sufficient.
- `repair-verified`: repair backend and verification passed.
- `needs-human`: human gate, ambiguous finding, repeated failure, or unsupported
  repair path.
- `failed`: review or repair failed in a trusted way.
- `invalid`: review or repair evidence is malformed or stale.

`repair-verified` applies only to the reviewed packet. It does not imply the
whole workflow is complete.

## Workflow Architecture

V2.5 uses these patterns:

- Sequential: V1 trust -> V2 trust -> review -> optional repair -> final status.
- Adversarial Verify: reviewers attempt to refute execution evidence and
  verification claims.
- Loop Until Dry: repair can repeat only under a strict retry cap; first release
  should cap at one repair.
- Human Gate: risky or ambiguous findings stop with preserved artifacts.
- Resume And Cache: trusted review/repair artifacts are skipped on resume.

## Fixtures

Required V2.5 fixture families:

- review approved from trusted V2 executed evidence,
- review request changes from trusted failed verification,
- review malformed JSON invalidation,
- review contract coherent tamper invalidation,
- review source attempt mismatch invalidation,
- repair blocked by V1 risk gate,
- repair blocked by unsupported risk category,
- repair prepared without backend,
- repair local fixture success,
- repair verification fail,
- retry cap stops with `needs-human`,
- optional fixture failure preserves keep only when required fixtures pass.

## Release Criteria

V2.5 is releasable when:

- V0.5, V1, and V2 release checks still pass.
- V2.5 self-test passes.
- V2.5 manifest records `decision: "keep"`.
- review and repair artifacts have parent-level contract ledgers.
- coherent tampering of review, repair, sidecar, and hash files invalidates
  resume.
- final status never claims full workflow completion.
- docs record the exact manifest command and generated summary.

The implemented first-loop keep gate is:

```bash
python scripts/execute_packet.py --manifest fixtures/v2.5/manifest.json --out out/v2.5/final
```

It proves deterministic review approval, change requests, review resume,
coherent review tamper invalidation, stale review invalidation after a new V2
attempt, replacement review append after that new attempt, stale repair
invalidation after a new review, repair prompt preparation, and non-actionable
repair refusal.

## V3 Handoff

V3 may consume only packets with trusted V2.5 terminal states:

- `review-approved`,
- `repair-verified`.

V3 must not advance from `failed`, `invalid`, `changes-requested`,
`repair-prepared`, `review-pending`, or `needs-human`, even when a
`--human-approved` flag is present. A later slice may define a stronger human
override contract.
